# 约个钓 — 后端设计方案 v1

> **日期**：2026-05-18
> **后端框架**：NestJS (TypeScript)
> **核心约定**：所有 HTTP 接口统一使用 **POST** 方法，RPC 风格组织路径
> **目标**：覆盖 v1 17 屏所需的全部 API，可在 4-8 周内由个人开发者完成

---

## 一、技术选型

### 1.1 推荐组合(主选)

| 维度 | 选型 | 选它的理由 |
|---|---|---|
| 框架 | **NestJS 10 + TypeScript 5** | 已确定 |
| 数据库 | **PostgreSQL 16 + PostGIS** | 内置地理空间索引(GiST)，附近钓点/钓友/组队全要 |
| ORM | **Prisma 5** | DX 最好，类型推导强；少量 PostGIS 查询用 `$queryRaw` |
| 缓存 | **Redis 7** | 宜钓指数、Feed 时间线、会话、限流 |
| 对象存储 | **腾讯云 COS** + CDN | 小程序原生 SDK 顺手；阿里 OSS 也可 |
| 内容安全 | **微信 msgSecCheck / imgSecCheck** | 文本图片审核，发布前同步调用 |
| 推送 | **微信订阅消息** subscribeMessage | 报名通过/新评论/组队提醒 |
| 任务队列 | **BullMQ** (基于 Redis) | 异步上传转码、定时计算宜钓指数 |
| 日志 | **Pino** + 控制台 | 高性能，结构化 |
| 配置 | **@nestjs/config** + `.env` | |
| 校验 | **class-validator + class-transformer** | NestJS 官方推荐 |
| API 文档 | **@nestjs/swagger** | 即使全 POST 也能生成清晰文档 |

### 1.2 数据库选型对比(如果你想看其它选项)

| 选项 | 优点 | 缺点 | 推荐度 |
|---|---|---|---|
| **PostgreSQL + PostGIS** | 地理查询最强、事务可靠、Prisma 友好 | 运维比 MySQL 略重 | ⭐⭐⭐⭐⭐ |
| MySQL 8 + Spatial | 国内云资源最便宜、运维熟 | GIS 函数比 PostGIS 残缺 | ⭐⭐⭐ |
| MongoDB | Schema 灵活、地理查询能用 | 事务弱、ORM 体验差 | ⭐⭐ |
| 微信云开发 | 零运维、自动鉴权 | 锁死微信生态，多端编译失去意义 | ⭐ |

**结论：选 PostgreSQL + PostGIS**。租阿里云 RDS for PostgreSQL 或腾讯云 TencentDB for PostgreSQL，1 核 2G 起步够 v1 用，月费 ~50-100 元。

### 1.3 部署建议(MVP 阶段)

```
[微信小程序/H5]
     │ POST(HTTPS)
     ▼
[Nginx 80/443]
     │
     ▼
[NestJS 服务 :3000] ──→ [Redis] (缓存/限流/队列)
     │
     ├──→ [PostgreSQL + PostGIS] (主库)
     ├──→ [腾讯云 COS] (图片直传)
     ├──→ [微信开放接口] (登录/订阅消息/内容安全)
     └──→ [和风天气 API] (天气数据)
```

单台 2C4G 云服务器(60-100 元/月) + RDS + Redis 即可起步。日活 5000 以下不用拆。

## 二、API 接口规范(**全 POST、RPC 风格**)

### 2.1 统一约定

- **方法**：一律 `POST`
- **路径**：`/api/{module}/{action}`，例如 `/api/spots/list`、`/api/catches/like`
- **入参**：JSON Body(`Content-Type: application/json`)
- **鉴权**：Header `Authorization: Bearer <jwt>`(登录、上传签名等少数接口免鉴权)
- **统一响应**：

  ```json
  {
    "code": 0,
    "msg": "ok",
    "data": { ... },
    "traceId": "abc123"
  }
  ```
  业务错误用非 0 `code` + `msg`；HTTP 状态码统一 200(除鉴权 401)；这样前端封装更简单。

- **分页约定**：cursor-based(更适合 Feed)，回 `{ list, nextCursor, hasMore }`

### 3.3 索引策略要点

1. **附近查询**走 `idx_spots_geom`(GiST)：`WHERE ST_DWithin(geom, ST_MakePoint(?,?)::geography, ?)`
2. **Feed 时间线**走 `idx_catches_feed`(部分索引，只索引已审核通过的)
3. **未读消息计数**走 `idx_notif_user_unread`
4. **关系表**(follow/like/favorite)用复合主键，天然唯一约束

### 3.4 容量预估(v1 一年)

| 数据 | 量级 | 注 |
|---|---|---|
| 用户 | 1-5 万 | 1-2 个种子城市 |
| 钓点 | 1-3 千 | UGC 上报，去重后 |
| 鱼获 | 10-50 万 | 主要数据增长项 |
| 评论 | 50-200 万 | 长尾 |
| 通知 | 100-500 万 | 加 TTL 30 天自动清理 |

→ PostgreSQL 单库轻松扛住，无需分库分表。

## 五、关键技术点设计

### 5.1 微信登录(`POST /api/auth/wx-login`)

```typescript
// 入参
{ code: string, encryptedData?: string, iv?: string }

// 后端流程
1. code → wx.code2Session → openid, unionid, session_key
2. SELECT user WHERE openid=? → 没有就创建占位用户(status='need_profile')
3. 签发 JWT (含 userId, openid)
4. 返回 { token, user, needProfile: boolean }

// 前端拿到 token 后存 storage，后续请求带 Authorization
```

`session_key` 暂存 Redis 30 分钟(`wx:sk:{userId}`)，用于后续解密 phone/avatar 数据。

### 5.2 附近钓点(`POST /api/spots/nearby`)

```typescript
// 入参
{
  lat: number, lng: number,
  radius?: number,          // 米，默认 5000
  type?: 'wild'|'black'|...,
  fishSpecies?: string[],
  limit?: number,
  cursor?: string
}

// 实现(Prisma + raw SQL)
await prisma.$queryRaw`
  SELECT id, name, type,
         ST_X(geom::geometry) as lng,
         ST_Y(geom::geometry) as lat,
         ST_Distance(geom, ST_MakePoint(${lng},${lat})::geography) as distance,
         fish_species, avg_rating, photos[1] as cover
  FROM spots
  WHERE status='approved'
    AND ST_DWithin(geom, ST_MakePoint(${lng},${lat})::geography, ${radius})
    ${type ? Prisma.sql`AND type=${type}` : Prisma.empty}
  ORDER BY distance ASC
  LIMIT ${limit}
`;
```

聚合气泡(`12` 标号)在前端按 zoom 计算，后端不做。

### 5.3 宜钓指数(`POST /api/weather/index`)

```typescript
// 入参 { lat, lng, date? }
// 流程
1. geohash = encodeGeohash(lat, lng, precision=5)
2. cacheKey = `weather:idx:${geohash}:${yyyymmddhh}`
3. cache hit → 直接返回
4. miss →
   a. 调和风天气拿 24h 预报
   b. 算法:
      score = 0.30 * pressureScore(气压变化率)
            + 0.20 * tempScore(舒适度)
            + 0.20 * windScore(风力 < 4 级)
            + 0.15 * precipScore(降水)
            + 0.15 * moonScore(月相)
      → 归一化 0-100
   c. 写入 Redis TTL 1h
   d. 返回 { score, factors: {...}, advice: '气压稳定，建议早晚窗口' }
```

算法独立成 `FishingIndexService`，方便 v2 加历史回归。

### 5.5 发布鱼获(`POST /api/catches/create`)

```typescript
// 入参
{
  photos: string[],          // 已经走过上传审核的 URL
  fishSpecies: string[],
  weightG?: number,
  lengthCm?: number,
  technique?: string,
  spotId?: number,
  lng?: number, lat?: number,
  content?: string,
  locationVisible: boolean,
  allowComments: boolean
}

// 流程
1. 校验用户已完善资料、未被封禁
2. 同步调 msgSecCheck(content) — 命中直接 400
3. 写库(review_status='approved'，因为图已审过，文本同步审过)
4. 异步任务:
   a. 抓当下天气写入 weather_snapshot
   b. 更新 user_stat.catches_count
   c. 给关注者发"动态"通知(批量入 notifications 表)
5. 返回 { catch }
```

### 5.6 Feed 排序(简单可靠版本)

```typescript
// /api/catches/feed - tab: 'follow' | 'nearby' | 'hot'

follow: 
  SELECT c.* FROM catches c
  JOIN follow f ON f.followee_id = c.user_id
  WHERE f.follower_id = ? AND c.review_status='approved'
  ORDER BY c.created_at DESC
  LIMIT 20

nearby:
  SELECT c.*, ST_Distance(...) as d FROM catches c
  WHERE review_status='approved' AND created_at > now()-'7 days'
  ORDER BY d + EXTRACT(EPOCH FROM now()-c.created_at)/3600 ASC
  LIMIT 20

hot:
  SELECT c.*,
    (like_count*2 + comment_count*3 + fav_count*1) AS score
  FROM catches WHERE created_at > now()-'7 days' AND review_status='approved'
  ORDER BY score DESC, created_at DESC
  LIMIT 20
```

v2 再上召回 + 精排。

### 5.7 组队报名审批(状态机)

```
                ┌── approve ──→ approved ─── attended (event 后)
                │                        └── no_show (event 后)
pending ────────┤
                ├── reject ───→ rejected
                └── quit ─────→ quit (申请者主动撤回)
```

发起人在 `end_time + 24h` 内可标记爽约，更新 `team_members.status='no_show'` 并触发 `user_stat.no_show_count++`，通过定时任务 (`@Cron`) 自动关闭过期组队。

### 5.8 内容审核

| 类型 | 时机 | 调用 |
|---|---|---|
| 文本(鱼获/评论/钓点描述) | 同步、提交前 | 微信 msgSecCheck v2 |
| 图片(头像/鱼获/钓点) | 异步、上传回调 | 微信 imgSecCheck |
| 钓点本身 | 人工审核 | 管理后台(v2) |

审核结果统一写 `content_review` 表，便于运营回查。

### 5.9 鉴权与白名单

- JWT 默认全局开启(`AuthGuard` 全局 Guard)
- 用 `@Public()` 装饰器标注免登路由(只有 `auth.wx-login`、`auth.refresh`、`common.config`)
- `@CurrentUser()` 装饰器从 request 注入当前用户
- Token 1h 过期，refresh token 30 天

### 5.10 限流

- 全局：每 IP 每分钟 60 请求 (`@nestjs/throttler`)
- 写操作：每用户每分钟 10 次(发鱼获、评论)
- 关键写：发钓点每天 5 次/人(防灌)
